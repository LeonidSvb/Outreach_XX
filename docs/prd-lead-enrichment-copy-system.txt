ЗАЧЕМ ЭТО ВООБЩЕ НУЖНО
--------------------------------------------------------------------------------
Сейчас копия генерируется напрямую из сырого контекста компании — AI каждый раз
сам выбирает ICP, pain, угол. Результат: разные письма, нет повторяемости,
невозможно понять что сработало.

Цель — сделать так, чтобы каждое письмо генерировалось из контролируемых
переменных: фиксированный ICP + фиксированный pain + стабильный шаблон.
Это превращает процесс отправки в систему тестирования гипотез.

Главный принцип:
  LLM извлекает факты
  Скрипты принимают решения
  LLM формулирует текст

================================================================================
ДАННЫЕ О ЛИДАХ
================================================================================

Концепция:
  Все данные о лиде агрегируются в одной таблице/представлении по ключу email.
  Источники: PlusVibe, Google Sheets, CSV, enrichment.

Что хранится:
  Базовые поля (из PlusVibe):
    email, first_name, last_name, company, title, domain,
    campaign_id, status, linkedin_url, и т.д.

  Enrichment поля (добавляются отдельно):
    website_summary       -- ~1000 символов, структурированное описание
    company_size          -- диапазон: 1-10, 10-50, 50-200, 200+
    linkedin_description  -- опционально, как вторичный сигнал

  Extraction поля (результаты LLM):
    company_types         -- список: ["recruiting", "saas", ...]
    services              -- список: ["exec_search", "staffing", "rpo", ...]
    primary_service       -- строка или null
    service_confidence    -- high / medium / low
    service_reasoning     -- короткое объяснение

    industries            -- список: ["manufacturing", "healthcare", ...]
    primary_industry      -- строка или null
    industry_confidence   -- high / medium / low

    target_roles          -- список: ["COO", "Plant Manager", ...]
    roles_confidence      -- high / medium / low
    icp_reasoning         -- откуда это взялось

    seniority_focus       -- exec / mid / mixed

  Словарные поля (после rules layer):
    resolved_icp          -- итоговый ICP для копии
    resolved_decision_makers -- итоговые роли
    resolved_pain         -- итоговый pain

  Мета:
    enriched_at, extraction_at, enrichment_source


================================================================================
LLM EXTRACTION — КАК РАБОТАЕТ
================================================================================

Что извлекаем:
  Два блока извлекаются через LLM из website_summary (+ опционально LinkedIn):

  Блок A — Company + Service:
    - Какие сервисы предоставляет компания (список)
    - Какой из них primary (если есть явный фокус)
    - confidence + reasoning

  Блок B — ICP Context (главный блок):
    - В каких индустриях работает
    - Какой industry primary
    - Какие роли упоминаются как их клиенты
    - confidence + reasoning
    - seniority focus

Правила confidence:
  high   -- явно написано на сайте, используются слова "specialize", "focus on",
            упоминается 2+ раза
  medium -- косвенно, можно догадаться
  low    -- предположение, мало данных

Если primary = null:
  Значит компания — дженералист, нет чёткого фокуса.
  Это тоже важная информация, фиксируется.

Reasoning:
  Короткая строка: "mentioned 3x", "no clear focus", "specialize in exec search"
  Нужен для дебага и понимания качества данных.


================================================================================
СЛОВАРИ (RULES LAYER)
================================================================================

Зачем:
  LLM может ошибаться или давать разные формулировки.
  Словари фиксируют: какой ICP, какие роли, какой pain использовать для
  каждого сегмента. Это убирает рандом и делает результаты сравнимыми.

Логика работы:
  1. Берём primary_service + primary_industry
  2. Ищем в словаре соответствующий сегмент
  3. Получаем: icp, decision_makers, pains
  4. Если confidence = medium/low → принудительно берём из словаря (fallback)
  5. Если confidence = high → используем данные из extraction

Где хранятся:
  В PostgreSQL, таблица dictionaries (или segments).
  Редактируются вручную или через Streamlit.

Структура сегмента в словаре:
  {
    "segment_id": "manufacturing_staffing",
    "primary_service": "staffing",
    "primary_industry": "manufacturing",
    "icp": "manufacturing companies",
    "decision_makers": ["Plant Manager", "Ops Director"],
    "pains": [
      "inconsistent hiring pipeline",
      "difficulty filling roles fast"
    ]
  }

Важно:
  Не более 2-3 ролей на сегмент.
  Не более 2-3 pain на сегмент.
  Сегменты определяются после анализа реальной базы лидов [TBD].


================================================================================
PAIN LIBRARY
================================================================================

Зачем:
  AI не выбирает pain самостоятельно каждый раз.
  Pain генерируются один раз (через LLM или вручную), отбираются, фиксируются.
  Дальше используются повторно без изменений.

Правила:
  - живой человеческий язык, не "corporate speak"
  - конкретно для индустрии/роли, а не общие слова
  - 2-3 варианта на сегмент для A/B теста

Хранение:
  В PostgreSQL, как часть словаря сегмента.

Сегменты и pain — определяются после анализа базы лидов [TBD].


================================================================================
COPY GENERATION
================================================================================

Базовая формула:
  "I'm around [ICP] daily and they keep saying [pain]"

  ICP = из resolved_icp (словарь или extraction)
  Pain = из resolved_pain (pain library)

Вариативность:
  Фиксировано: ICP, pain, структура
  Может меняться: wording, тон (casual/direct), длина

A/B тест:
  Одни и те же лиды → разные варианты pain или формулировки
  Разные сегменты → разные копии

Экспорт:
  - CSV для загрузки в PlusVibe вручную (сейчас)
  - Позже — через PlusVibe API (создание кампаний, A/B варианты)

Хранение копий:
  Таблица copies в PostgreSQL:
    id, type, version, content, variables (jsonb), active, created_at

Версионирование:
  Каждый вариант копии сохраняется.
  Привязка: копия → кампания → результаты.


================================================================================
МЕТРИКИ
================================================================================

Минимальный трекинг:
  - reply rate
  - positive reply rate

Связь:
  Каждый reply привязан к: лиду → копии → сегменту → кампании

Это позволяет видеть:
  Какой сегмент + какой pain → дал какой reply rate
  Где ICP mismatch (много "not relevant" → не попали в pain)

Расширенная аналитика — отдельный блок [TBD, ~10-15 метрик].


================================================================================
СЛОЙ КЛАССИФИКАЦИИ РЕПЛАЕВ
================================================================================

Зачем:
  PlusVibe имеет свою классификацию — она поверхностная и не всегда корректная.
  Нужен собственный слой поверх.

Что классифицируем:
  sentiment:        positive / neutral / negative
  intent:           interested / not_interested / objection / unsubscribe
  objection_type:   timing / budget / no_need / wrong_persona /
                    already_solved / implicit_rejection / unknown
  quality_signal:   high / medium / low

Где хранится:
  Таблица reply_analysis в PostgreSQL.
  Поле source: "plusvibe" (исходная) vs "custom" (наша).

Это основа для evaluation loop (Проект 2).


================================================================================
STREAMLIT — ВНУТРЕННИЙ UI
================================================================================

Назначение:
  Не для клиентов. Только для личного использования.
  Управление системой без написания скриптов каждый раз.

Минимальный функционал:
  - просмотр лидов + их enrichment данных
  - ручное редактирование словарей и сегментов
  - запуск генерации копий (с выбором сегмента / pain)
  - просмотр версий копий
  - базовая статистика по кампаниям

Когда делать:
  После того как enrichment pipeline работает.
  Не раньше.
