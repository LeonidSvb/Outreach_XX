ЗАЧЕМ ЭТО НУЖНО
--------------------------------------------------------------------------------
Ручной анализ кампаний — это перегруз: смотришь цифры, пытаешься найти паттерн,
устаёшь, уходишь заниматься чем-то другим. Результат — нет накопления знаний,
нет итераций.

Цель — сделать агента, который берёт данные из Postgres, сам находит паттерны,
сам предлагает (и со временем сам делает) улучшения копий и A/B тесты.

Главный принцип:
  Агент не делает ничего магического — он просто делает то, что ты бы делал
  руками, только быстрее и не устаёт.


================================================================================
ЧТО УМЕЕТ АГЕНТ
================================================================================

1. Аналитика
   - Читает данные из PostgreSQL (кампании, реплаи, метрики, копии, сегменты)
   - Находит паттерны: какой сегмент работает, какой нет
   - Видит: какой pain дал reply, какой — "not relevant"
   - Сравнивает версии копий между собой
   - Выдаёт текстовый отчёт: что работает, что убивать

2. Предложение улучшений копии
   - На основе данных предлагает твики: subject line, hook, pain, тон
   - Либо берёт из библиотеки копий и адаптирует под сигнал
   - Не пишет с нуля — работает с существующей структурой

3. A/B тест-менеджмент
   - Создаёт варианты копии (A, B, C...)
   - Разбивает пул лидов на батчи
   - Загружает в PlusVibe как отдельные кампании или варианты
   - Ждёт накопления данных (триггер: N отправлено / N реплаев / ручной старт)
   - Сравнивает результаты
   - Убивает слабый вариант
   - Создаёт следующий вариант против победителя
   - Итерирует

4. Управление лидами для кампаний
   - Берёт пул лидов из Postgres по фильтру (сегмент, статус, и т.д.)
   - Разбивает на батчи (например, 2000 лидов → 4 батча по 500)
   - Назначает каждому батчу копию / вариант


================================================================================
ИНСТРУМЕНТЫ АГЕНТА (TOOLS)
================================================================================

  get_campaign_stats       -- получить статистику кампании из Postgres
  get_replies              -- получить реплаи с классификацией
  get_copies               -- получить версии копий из библиотеки
  get_leads_segment        -- получить лидов по фильтру (сегмент, статус и т.д.)
  save_copy_version        -- сохранить новую версию копии
  create_campaign          -- создать кампанию в PlusVibe (через API)
  add_leads_to_campaign    -- загрузить батч лидов в кампанию
  compare_ab_results       -- сравнить два варианта A/B по метрикам

Агент НЕ имеет права:
  - физически удалять данные из Postgres
  - менять enrichment данные лидов
  - запускать email рассылку без подтверждения (на старте)


================================================================================
ЛОГИКА A/B ЦИКЛА
================================================================================

  Шаг 1. Старт
    Есть пул лидов + 2 варианта копии (A и B)
    Агент делит пул на 2 батча
    Загружает в PlusVibe как кампанию A и кампанию B

  Шаг 2. Накопление данных
    Триггер: вручную (ты говоришь "проверь")
    Позже: по крону или по N отправленных

  Шаг 3. Сравнение
    Агент смотрит: reply rate, positive rate, objection breakdown
    Определяет победителя

  Шаг 4. Итерация
    Слабый вариант — убивается (кампания останавливается)
    Победитель остаётся как базовый
    Агент предлагает вариант C (твик победителя)
    Цикл повторяется

  Шаг 5. Выход
    Когда найден стабильно работающий вариант → фиксируется в библиотеке копий
    Сегмент помечается как "validated"


================================================================================
КАК ВЫГЛЯДИТ ОБЩЕНИЕ С АГЕНТОМ
================================================================================

Запуск — вручную через Claude Code:
  "проанализируй кампании за последние 2 недели"
  "сделай A/B тест для сегмента manufacturing_staffing"
  "какой pain лучше всего заходит в exec search?"

Агент:
  - читает данные
  - делает анализ
  - выдаёт отчёт или список действий
  - спрашивает подтверждение перед созданием кампании
  - запоминает контекст внутри сессии

Позже (если всё работает):
  - cron запуск раз в неделю
  - отчёт в Telegram (после добавления уведомлений)

================================================================================
ДАННЫЕ КОТОРЫЕ ИСПОЛЬЗУЕТ АГЕНТ
================================================================================

Читает из Postgres:
  - campaigns            -- список кампаний + статус
  - emails               -- отправленные письма
  - replies              -- реплаи + классификация
  - reply_analysis       -- наша кастомная классификация
  - copies               -- библиотека копий и версий
  - leads_master         -- лиды + enrichment данные
  - dictionaries         -- словари сегментов

Пишет в Postgres:
  - copies               -- новые версии копий
  - ab_tests             -- трекинг A/B тестов (какой вариант против какого)
  - agent_log            -- лог действий агента (что сделал, почему)


================================================================================
МЕТРИКИ ДЛЯ СРАВНЕНИЯ ВАРИАНТОВ
================================================================================

Основные (для A/B решения):
  - reply_rate            -- % ответивших от отправленных
  - positive_rate         -- % положительных от отправленных
  - objection_breakdown   -- топ причины отказов

Дополнительные (для анализа):
  - сегмент vs reply_rate
  - pain vs positive_rate
  - decision_maker_role vs ответ
